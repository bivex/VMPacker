 # VMPacker: Hybrid Mode Plan

 ## 1. Концепция (Concept)
 Гибридный режим (Hybrid Mode) позволяет исполнять часть кода внутри
 виртуальной машины (VM), а часть — в виде нативных машинных инструкций
 (Native), переключаясь между ними «на лету». Это дает два главных
 преимущества:
 1.  **Производительность:** Тяжелые математические вычисления (FPU,
 циклы) могут выполняться без оверхеда стековой машины.
 2.  **Усложнение анализа:** Смешивание нативного кода с байт-кодом
 сильно ломает автоматические анализаторы и декомпиляторы.

 ## 2. Архитектурное решение (Architecture)
 Поскольку весь payload (включая байт-код) мапится в память с правами
 **RX** (`PF_R | PF_X`), байт-код сам по себе является исполняемым!

 Мы добавили новую инструкцию VM: `OpNativeExec`.

 ### Формат инструкции:
 `[OpNativeExec (1 байт)] [Size uint16 (2 байта)] [ ... Native Machine Code ... ] [ Нативный RET ]`

 ### Как это работает:
 1.  **Translator (Go):** При анализе инструкции транслятор случайным
 образом (или по эвристике) решает не переводить ее в стек-операции, а
 оставить нативной (только безопасные инструкции: ALU, FPU, MOV, без
 переходов). Он копирует байты оригинальной инструкции, добавляет
 инструкцию возврата (`RET`) и оборачивает это в `OpNativeExec`.
 2.  **Interpreter (C/ASM):** Хендлер `h_native_exec` делает следующее:
     *   Выгружает контекст VM (`vm->R`) в реальные регистры процессора.
     *   Делает `CALL` на адрес внутри байт-кода (где лежат нативные байты).
     *   Нативная инструкция исполняется реальным процессором.
     *   Нативный `RET` возвращает управление хендлеру.
     *   Хендлер сохраняет реальные регистры обратно в `vm->R`.
     *   `vm->pc += 3 + Size + SizeOf(RET)`.

 ## 3. Реализация (Implementation)

 ### Шаг 1: Определение Opcode [Готово]
 *   Добавлен `OpNativeExec` (`0x49`) в `pkg/vm/opcodes.go`.
 *   Добавлен `OP_S_NATIVE_EXEC` в заголовочные файлы интерпретаторов (`stub/linux/x86_64/vm_opcodes.h`, `stub/linux/arm64/vm_opcodes.h`).
 *   Определен логический ID `OP_ID_SNATIVEEXEC 141` в `vm_opcodes_dynamic.h` для обеих архитектур, обновлен `OP_ID_COUNT` до 142.
 *   Обновлена Go-таблица `isa.go` (OpIdSNativeExec) и `opPtrs`.

 ### Шаг 2: Реализация Context Switch [Готово]
 *   Реализован хендлер `h_native_exec` в `stub/linux/x86_64/vm_handlers/h_system.h` и `stub/linux/arm64/vm_handlers/h_system.h`.
 *   Для x86_64 используется `register ... asm()` для быстрого сохранения/восстановления регистров и `call *reg`.
 *   Для AArch64 — аналогично с использованием x0-x29, lr, sp alignment.
 *   Добавлены обёртки `hw_native_exec` в `vm_dispatch.h` для обеих архитектур.
 *   Включены в таблицу инициализации `vm_init_jump_table`: `tbl[OP_ID_SNATIVEEXEC] = hw_native_exec;`.
 *   Обновлены `vm_logical_insn_size` в `vm_decode.h` (оба архитектуры): возвращает `4` (минимальный размер) для нового opcode.

 ### Шаг 3: Поддержка в Translator [Готово]
 *   Добавлено поле `hybrid` в `Translator` (`pkg/arch/x86_64/translator.go`).
 *   Добавлен метод `SetHybrid(enabled bool)`.
 *   Реализован `tryEmitNative(raw []byte, inst x86asm.Inst) bool`: проверяет безопасность инструкции, 20% шанс, эмитирует `OpNativeExec` + длина + raw-байты + `RET`.
 *   В `Translate()` добавлен вызов `tryEmitNative` перед обычной трансляцией.
 *   Для ARM64 аналогичная реализация не требуется, так как транслятор для ARM64 пока не поддерживает Hybrid Mode (отметка TODO).

 ### Шаг 4: Тестирование [Выполнено]
 *   Go-тесты пакета `pkg/vm` проходят успешно.
 *   Тесты `pkg/arch/x86_64` проходят успешно.
 *   C-стабы требуют кросс-компиляцию (x86_64 на arm64 хост не поддерживается). Код готов к сборке на цели x86_64/ARM64.

 ## 4. Ожидаемые улучшения
 *   Интеграция флага `-hybrid` в CLI утилиту `vmpacker`.
 *   Тонкая настройка вероятности эмиссии нативного кода (параметр).
 *   Безопасность: валидация нативных кусков на отсутствие опасных инструкций (интеллектуальный анализ).
 *   Расширение Hybrid Mode на архитектуру ARM64 (транслятор).
