# Architecture Task 1: Clang & FPU/SIMD Compatibility

This document describes the foundational work performed to support ARM64 code generated by modern Clang/NDK compilers, which heavily utilize floating-point registers and complex stack machine patterns.

## 1. Accomplishments (What we did)

### Core Architecture Overhaul
*   **FPU/SIMD Support**: Added 32 internal 128-bit vector registers (`V0-V31`). Implemented handlers for arithmetic (`FADD`, `FMUL`, `FMAX`, `FMIN`), conversions (`FCVTZU`, `SCVTF`, `FCVT`), and data movement (`FMOV`, `LDP/STP` for Q/D registers).
*   **Stack Machine Stabilization**: Fully transitioned ALU operations to a Stack Machine (`eval_stk`). 
    *   **Fixed Push/Pop Order**: Store handlers now correctly pop `value` then `address`.
    *   **Fixed Pointer Integrity**: Reconciled `SPUSH` (post-increment) and `SPOP` (pre-decrement) to ensure index 0 is used correctly.
*   **AArch64 Strict Alignment**: Refactored `vm_ctx_t` to ensure `V` registers, `eval_stk`, and the memory stack (`vm_stk`) are **16-byte aligned**. This is mandatory for AArch64 `LDP/STP` instructions.
*   **XZR/SP Separation**: Mapped the Zero Register (XZR) to internal index `63` to permanently resolve conflicts with the Stack Pointer (SP, index `31`).

### Environment & Tooling
*   **vmp-builder Docker**: Optimized the build container to include `gdb` and `binutils`. This allows instant debugging without installing tools at runtime.
*   **Stub Hardening**: Replaced C-based `memcpy` with a volatile-safe version (and experimented with inline ASM) to prevent the compiler from generating recursive calls in the `-nostdlib` environment.

## 2. Debugging Workflow (How to fix crashes)

If the virtualized binary crashes (Segmentation Fault), follow these steps:

### Step A: Capture the crash in GDB
Use the provided `debug_demo.sh` or run:
```bash
docker run --rm --platform linux/arm64 -v $(pwd):/work -w /work/demo vmp-builder \
    gdb -batch -ex 'run' -ex 'bt' -ex 'info registers' ./demo_complex.vmp
```
*   **Look for the PC**: e.g., `0x0000000000428780`.
*   **Identify the offset**: Payload base is usually `0x420000`. So offset = `0x8780`.

### Step B: Map offset to VM Handler
Disassemble the debug ELF of the stub:
```bash
docker run --rm --platform linux/arm64 -v $(pwd):/work -w /work/build/stub vmp-builder \
    aarch64-linux-gnu-objdump -d -S vm_interp.elf | grep -A 20 -B 10 "8780"
```
*   This will show you exactly which C function (e.g., `h_s_st64`) or instruction is failing.

### Step C: Verify Math/Logic
If it runs but returns `0`, check the FPU conversion handlers (`h_fcvt_fi`, `h_fcvt_if`) in `stub/linux/arm64/vm_handlers/h_fpu.h`. Ensure the `type` byte correctly passes the `unsigned` and `sf` bits.

## 3. Future Work (What needs to be done)

1.  **SIMD Advanced Ops**: Currently only basic scalar FPU and wide memory moves are implemented. Full NEON vector math (e.g., `ADD V0.4S, V1.4S, V2.4S`) is not yet supported.
2.  **Instruction Reversal**: The logic for backward execution (`reverse = 1`) is implemented but should be thoroughly stress-tested with `demo_complex`.
3.  **Optimization**: The `SPOP/SPUSH` macros add overhead. For high-performance code, consider merging multiple stack ops into specialized "super-ops".
4.  **ARM32 Parity**: Port these FPU/SIMD fixes back to the ARM32 stub (`stub/arm32/`) if Android 32-bit support is still required.

---
*Note: The system is currently tuned for Clang -O1/-O2 output. If adding support for -O3, watch out for complex auto-vectorization patterns.*
